## Управление пользователями

### Создание суперпользователя (Директор)

Для создания нового суперпользователя с правами директора используйте безопасный CLI-скрипт:

```bash
cd backend
npm run create-superuser
```

Скрипт интерактивно запросит:
- **Email** (используется как логин)
- **Пароль** (минимум 8 символов)
- **Подтверждение пароля**
- **Полное имя**

Пароль будет автоматически зашифрован с использованием bcrypt перед сохранением в базу данных.

#### Пример использования

## Требования к паролям

### Минимальные требования

- **Длина:** минимум 8 символов
- **Рекомендуется:**
  - Комбинация букв верхнего и нижнего регистра
  - Цифры
  - Специальные символы (!@#$%^&*)

### Примеры надежных паролей

✅ Хорошие пароли:
- `MySecure2024Pass!`
- `Tr@nsp0rt#Admin`
- `D!rect0r_TL196`

❌ Слабые пароли:
- `password`
- `12345678`
- `admin123`

## Хранение паролей

### Технология

Система использует **bcrypt** для хэширования паролей с параметрами:
- **Алгоритм:** bcrypt
- **Salt rounds:** 10
- **Формат хэша:** `$2b$10$...` (60 символов)

### Безопасность

- Пароли **никогда** не хранятся в открытом виде
- При каждом хэшировании генерируется уникальная соль
- Невозможно восстановить исходный пароль из хэша
- Защита от rainbow table атак

## Изменение пароля существующего пользователя

### Через интерфейс (для директора)

1. Войдите как директор
2. Откройте раздел "Пользователи"
3. Найдите нужного пользователя
4. Нажмите "Редактировать"
5. Введите новый пароль
6. Сохраните изменения

Пароль будет автоматически зашифрован перед сохранением.

### Через базу данных (экстренный случай)

Если потеряли доступ к аккаунту директора:

```bash
# 1. Создайте хэш нового пароля
cd backend
node -e "const {hashPassword} = require('./utils/passwordUtils'); hashPassword('НОВЫЙ_ПАРОЛЬ').then(console.log).catch(console.error)"

# 2. Скопируйте полученный хэш (начинается с $2b$10$...)

# 3. Обновите пароль в БД
psql -U <username> -d <database> -c "UPDATE users SET password_hash = 'СКОПИРОВАННЫЙ_ХЭШ' WHERE email = 'admin@example.com';"
```

## Миграция существующих пользователей

### Если в базе есть пользователи с незашифрованными паролями

⚠️ **ВНИМАНИЕ:** После внедрения bcrypt старые пользователи с plain text паролями не смогут войти!

#### Вариант 1: Удалить и создать заново (рекомендуется)

```bash
# Удалить старого пользователя
psql -U <username> -d <database> -c "DELETE FROM users WHERE email = 'старый_email';"

# Создать нового с bcrypt
npm run create-superuser
```

#### Вариант 2: Сбросить пароли существующим пользователям

Попросите каждого пользователя обратиться к директору для сброса пароля через интерфейс.

## Best Practices

### Для администраторов

1. ✅ Используйте уникальные сложные пароли для каждого пользователя
2. ✅ Регулярно проверяйте список активных пользователей
3. ✅ Деактивируйте неиспользуемые аккаунты
4. ✅ Используйте осмысленные email-адреса в качестве логинов
5. ❌ Никогда не передавайте пароли в открытом виде
6. ❌ Не используйте одинаковые пароли для разных пользователей

### Для пользователей

1. ✅ Не сохраняйте пароли в незащищенных местах
2. ✅ Выходите из системы после завершения работы
3. ✅ Сообщайте администратору о подозрительной активности
4. ❌ Не делитесь своим паролем с другими
5. ❌ Не используйте слишком простые пароли

## Аудит безопасности

### Проверка хэшированных паролей

Чтобы убедиться, что все пароли надежно зашифрованы:

```bash
psql -U <username> -d <database> -c "SELECT email, password_hash FROM users;"
```

Все `password_hash` должны начинаться с `$2b$10$` и иметь длину 60 символов.

### Проверка активных пользователей

```bash
psql -U <username> -d <database> -c "SELECT email, full_name, is_active, last_login FROM users ORDER BY last_login DESC;"
```

## Восстановление доступа

Если потеряли доступ к системе:

1. **Есть доступ к базе данных:**
   - Используйте метод изменения пароля через БД (см. выше)

2. **Нет доступа к БД:**
   - Обратитесь к системному администратору сервера
   - Администратор может создать нового суперпользователя через скрипт

3. **Полная потеря доступа:**
   - Может потребоваться восстановление из резервной копии
